# Typing Projects Showcase

Welcome to this collection of typing-related web applications, including a feature-rich typing test client and a real-time typing battle server.

This repository uses a multi-service Docker environment to run all the necessary components, including frontends, backends, and databases.

## Projects Included

*   **`monkeytype-master`**: A local Monkeytype reference checkout used for UI and gameplay inspiration. It is not deployed as part of this platform.
*   **`typebattle`**: A Node.js backend server designed for real-time multiplayer typing races. It uses WebSockets for communication and is intended to be connected to a PostgreSQL database.
*   **`nfc_attendance`**: A Python-based attendance system that appears to use NFC technology. It is served using Gunicorn.
*   **Other Services**: The environment also includes `PostgreSQL` for the database and `Caddy` as a reverse proxy to manage traffic to the different services.

## Shared-platform migration

The attendance application continues to use its existing JSON/CSV files so the
kiosk and reports remain compatible. Successful registrations and attendance
scans are now mirrored to PostgreSQL (`school_students`, `nfc_cards`, and
append-only `attendance_events`) for the gradual move to a shared student
identity with TypeBattle. TypeBattle already uses the same PostgreSQL instance
and can join its accounts to a student through `student_id`.

ESP32 devices should use `POST /api/scan` with a JSON body containing `uid` and
a unique `scan_id`; the latter makes Wi-Fi retries idempotent. Set
`ESP32_API_KEY` before exposing attendance outside the local network and send
it as the `X-ESP32-Key` header. The legacy GET scan endpoint remains supported
during firmware migration.

If the PostgreSQL volume already exists, apply the one-time schema migration
before deploying this version:

```bash
docker compose exec -T postgres psql -U app -d school_platform < db/migrations/001_shared_school_identity.sql
```

## Quick Start

This setup is designed to be simple to run. From the root `Projects` directory, follow these steps.

### 1. Create Environment File

If you haven't already, copy the example environment file. This file contains the necessary secrets and passwords for the services, including the database.

For local development:

```bash
cp .env.example .env
```

For public deployment, use the production example and fill in your real domain names and secrets:

```bash
cp .env.production.example .env
```

Then edit `.env` and set the following values at minimum:

- `POSTGRES_PASSWORD`
- `ADMIN_PIN`
- `SESSION_SECRET`
- `ATTENDANCE_HOST`
- `TYPEBATTLE_HOST`
- `PUBLIC_SERVER_URL`
- `ESP32_API_KEY` (required before exposing ESP32 scan endpoints)


**Important**: The default password in `.env.example` may not match what your application code expects. Open the `.env` file and ensure the `POSTGRES_PASSWORD` matches the password used in your application's database connection string.

For example, based on the `how2start.md` file, the `typebattle` server expects the password `Angprey2026`. You should set it in your `.env` file like this:

```env
# .env

# ... other variables
POSTGRES_USER=app
POSTGRES_PASSWORD=Angprey2026
POSTGRES_DB=school_platform
# ... other variables
```

### 2. Build and Run

Use `docker-compose` to build the images and start all the services in detached mode (`-d`). The `--build` flag ensures that any recent code changes are included.

```bash
docker-compose up --build -d
```

### 3. Accessing the Services

Once the containers are running, the services will be available at the following local URLs, routed by the Caddy reverse proxy:

*   **Monkeytype/TypeBattle Frontend**: `https://type.localhost`
*   **NFC Attendance System**: `https://attendance.localhost`

Caddy automatically handles HTTPS for local development. You may need to accept a self-signed certificate in your browser the first time you visit the sites.

## Troubleshooting

### Database Connection Error

If you see `password authentication failed for user "app"` in the logs for the `typebattle` or `nfc_attendance` services, it means there is a mismatch between the `POSTGRES_PASSWORD` in your `.env` file and the password expected by the application.

1.  **Stop the services**: `docker-compose down`
2.  **Check the password**: Ensure the `POSTGRES_PASSWORD` in the `.env` file is correct.
3.  **Check application code**: Verify the database connection string in your `typebattle` and `nfc_attendance` code is using the correct environment variable or hardcoded password.
4.  **Restart**: `docker-compose up --build -d`

## Gemini Online Model (optional)

The attendance app uses an external Gemini-style model for AI chat. Set the following environment variables to enable Gemini:

```bash
export GEMINI_API_URL="https://your-gemini-endpoint.example/v1/generate"
export GEMINI_API_KEY="YOUR_API_KEY_OR_OAUTH_BEARER_TOKEN"
```

How it works:
- The Flask endpoint `/api/chat/assistant` sends the composed prompt to the configured Gemini URL with `Authorization: Bearer <GEMINI_API_KEY>`.
- If Gemini is not configured, the endpoint returns an error explaining that `GEMINI_API_URL` and `GEMINI_API_KEY` must be set.

Quick test (direct Gemini endpoint):

```bash
curl -s -X POST "$GEMINI_API_URL" \
	-H "Content-Type: application/json" \
	-H "Authorization: Bearer $GEMINI_API_KEY" \
	-d '{"prompt":"Hello","model":"gemini-3.1-flash-lite","stream":false}'
```

Quick test (app proxy):

```bash
curl -s -X POST http://127.0.0.1:5000/api/chat/assistant \
	-H 'Content-Type: application/json' \
	-d '{"message":"Hello, how many students?"}'
```

If you see a JSON reply with `reply` set, the integration is working.
